# TESTS.md

## 1. 문서 목적

이 문서는 `칭찬해줘`의 기능·디자인·안전·플랫폼 회귀를 방지하기 위한 검증 계약이다.
모든 구현 작업은 아래 자동 테스트와 수동 QA를 통과해야 완료로 간주한다.

## 2. 기본 명령

```bash
npm test
npm run build
npm run dev
```

기대 결과:

- `npm test`: 전체 테스트 통과
- `npm run build`: TypeScript `tsc --noEmit` 및 Vite production build 통과
- `npm run dev`: 로컬에서 정상 렌더링

작업 전과 작업 후에 모두 실행한다.

## 3. 자동 테스트 필수 계약

### 3.1 첫 방문 흐름

- Landing이 한국어 기본값으로 렌더링된다.
- 대상 확인 CTA가 동작한다.
- 칭찬 카드가 처음 3개만 보인다.
- `다른 한 줄 더 보기` 후 최대 5개가 보인다.
- 선택 전 다음 CTA가 비활성 또는 진행 불가다.
- 선택 후 다음 단계로 이동한다.

### 3.2 수정 흐름

- 문구를 수정하지 않아도 진행할 수 있다.
- 수정 시작 이벤트는 한 세션에서 한 번만 발생한다.
- caution 문구는 안내 후 저장 가능하다.
- caution 저장 시 caution 이벤트와 rewrite 저장 이벤트가 모두 발생한다.
- blocked 문구는 저장 불가다.
- blocked 상태에서 rewrite 저장 이벤트가 발생하지 않는다.

### 3.3 시간 저장 및 미리보기

- 시간 기본값이 유효한 `HH:mm` 형식이다.
- 저장 후 previewText가 채워진다.
- 첫 세션은 실제 알림 없이 미리보기 단계에서 멈춘다.
- notifications adapter 결과가 `preview_only`다.
- 실제 notification SDK 호출이 없다.

### 3.4 localStorage

- 정상 상태가 새로고침 후 복구된다.
- locale가 별도 키로 유지된다.
- corrupted JSON은 앱을 깨뜨리지 않고 기본 상태로 복구된다.
- 오래된/누락된 필드가 있어도 normalize 로직이 안전하게 처리한다.
- `sessionPhase: reopened`이면 체크인 단계로 진입한다.

### 3.5 체크인

현재 구현 카피 또는 개편 카피에 맞춰 동작을 검증한다.

- 세 선택지가 모두 동작한다.
- 각 선택의 이벤트가 정확히 한 번 발생한다.
- 사용자가 흐름을 닫거나 다시 선택할 수 있다.
- 결과 화면은 dismiss 가능하다.
- 가격, 할인, 결제 오해 문구가 없다.

### 3.6 i18n

- 한국어 기본 렌더링
- English 전환 가능
- 영어 앱명은 `Praise Me`
- 주요 화면의 문자열이 raw text로 하드코딩되지 않는다.
- ko/en 모두 버튼이 overflow되지 않는다.

### 3.7 analytics sanitizer

- 허용 필드만 유지한다.
- 자유 입력 원문은 제거한다.
- `text`, `message`, `rewriteText`, `previewText` 등 text-shaped key는 drop한다.
- 사용자 문장 일부가 event payload에 섞이지 않는다.

### 3.8 platform adapters

- auth는 anonymous
- payment entitlement는 false
- ads는 disabled
- analytics는 disabled/no-op
- notifications는 preview-only
- 실제 외부 SDK import가 live entrypoint에 없다.

## 4. 신규 v0.4 테스트 계약

재방문 홈·보관함·기록을 구현할 경우 다음 테스트를 추가한다.

### 4.1 재방문 홈

- 저장 상태가 있으면 홈에 오늘의 한 줄이 표시된다.
- 다음 확인 시간이 표시된다.
- `한 줄 바꾸기`가 선택 화면으로 이동한다.
- 저장 상태가 없으면 적절한 empty state가 나온다.

### 4.2 감정 결

- `잘 버텼어 / 잠깐 쉬어 / 다시 해보자 / 괜찮아`가 올바른 카탈로그를 연다.
- 각 결은 i18n key를 가진다.
- 선택 상태는 색상 외에도 체크/테두리로 구분된다.

### 4.3 보관함

- 도움이 된 한 줄을 저장할 수 있다.
- 같은 항목 중복 저장 정책이 일관적이다.
- 삭제할 수 있다.
- 삭제 후 UI와 localStorage가 일치한다.
- 빈 보관함 상태가 표시된다.

### 4.4 나를 챙긴 기록

- 방문하지 않은 날을 실패로 표시하지 않는다.
- 최근 7일 상태가 timezone 오류 없이 계산된다.
- 기기 시간 변경 시 앱이 깨지지 않는다.

## 6. 시각 QA

### 색상

- 주 색상은 라벤더 계열로 일관된다.
- 피치·민트·옐로·라일락은 보조 패널에만 사용한다.
- 배경과 카드의 대비가 충분하다.
- 경고/차단 색상이 브랜드 색상과 혼동되지 않는다.

### 깊이와 그림자

- 2.5D 깊이는 낮고 부드럽다.
- 그림자가 카드 경계를 흐리지 않는다.
- 모든 카드가 동일한 강도로 떠 있지 않는다.
- 히어로와 주요 CTA만 가장 높은 elevation을 가진다.

### 패널 계층

권장 우선순위:

1. 오늘의 한 줄 히어로
2. 주요 CTA
3. 감정 결 타일
4. 체크인/보관함
5. 보조 기록

- [ ] 같은 화면의 모든 패널이 동일한 크기와 강조를 가지지 않는다.
- [ ] 정보가 많은 패널은 배경색을 최소화한다.
- [ ] 2열 타일은 360px에서도 충분한 터치 영역을 갖는다.

### 타이포

- [ ] 화면 제목과 본문 계층이 명확하다.
- [ ] 본문 15px 미만 사용을 피한다.
- [ ] 한국어 줄바꿈이 어색하지 않다.
- [ ] 영어가 버튼 밖으로 넘치지 않는다.

### 모션

- [ ] `prefers-reduced-motion`을 고려한다.
- [ ] 카드 선택 애니메이션은 150–220ms 내외다.
- [ ] 과도한 bounce, confetti, 자동 반복 애니메이션이 없다.

## 7. 접근성 QA

- 모든 버튼에 명확한 접근 가능한 이름이 있다.
- 카드 전체가 버튼 역할이라면 semantic button 또는 radio 구조를 사용한다.
- focus-visible 스타일이 있다.
- 색상만으로 상태를 전달하지 않는다.
- 터치 영역은 최소 44×44px다.
- 텍스트 대비는 WCAG AA 수준을 목표로 한다.
- 언어 전환 버튼의 현재 선택 상태가 스크린리더에 전달된다.
- 입력 오류와 safety 안내가 `aria-live` 또는 적절한 설명 연결을 가진다.

## 8. 안전 QA

### caution 테스트 예

```text
난 왜 맨날 이래
한심하게 또 미뤘네
넌 왜 이것도 못 하니
```

기대:

- 부드러운 안내 표시
- 저장 가능
- 외부 전송 없음

### blocked 테스트 예

```text
죽어
자해하고 싶어
폭력으로 해결하자
이 앱이 진단하고 치료해줘
```

기대:

- 저장 불가
- 명확한 안내
- 원문 analytics 포함 금지

### 회귀 위험

- 단순 keyword 시스템을 전문 moderation처럼 표현하지 않는다.
- 앱 UI에서 치료·진단을 제공한다고 주장하지 않는다.

## 9. 네트워크 및 개인정보 QA

현재 MVP는 local-first다.

- 전체 첫 방문 흐름에서 불필요한 network call이 없어야 한다.
- 사용자 자유 입력 원문이 console, analytics, URL, query string에 노출되지 않아야 한다.
- localStorage 외 별도 저장을 하지 않는다.
- 외부 이미지·폰트·SDK 추가 시 개인정보/네트워크 영향을 검토한다.

## 11. 테스트 실패 처리

테스트가 실패하면 다음 순서로 대응한다.

1. 실패를 재현한다.
2. 기존 계약이 잘못된 것인지 구현이 깨진 것인지 판단한다.
3. 제품 결정을 바꾸지 않고 테스트만 삭제하지 않는다.
4. 계약 변경이 필요하면 `GOAL.md`와 `PLAN.md`의 근거를 먼저 갱신한다.
5. 수정 후 전체 테스트와 빌드를 다시 실행한다.
6. 세션로그에 원인과 해결을 기록한다.

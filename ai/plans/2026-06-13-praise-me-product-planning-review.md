# 칭찬해줘 상품성/기획성 리뷰

작성일: 2026-06-13
상태: review-complete
관점: Superpowers brainstorming + gstack CEO/founder review + design/eng/DX lens

## 한 줄 결론

`칭찬해줘`는 지금 방향이 맞다. 다만 상품성의 핵심은 "문장 앱"이 아니라
**3초 안에 좋은 목소리로 지금 듣고 싶은 말을 들려주는 오디오 감정 회복 앱**으로 잡아야 한다.

현재 구현된 5버튼 + `releasePack` + `audioManifest` 구조는 MVP로 적절하다.
다음에 기능을 넓히기보다, 실제 오디오 25개와 문장 품질 검수에 집중하는 것이 맞다.

추가 UX 검토 결과, 상위 버튼을 누른 뒤 다음 페이지에서 5개 오디오를 고르게 하는 흐름은
v0.1 메인 흐름으로 적용하지 않는다. 이 앱의 첫 약속은 탐색이 아니라 즉시 재생이다.

## 현재 확정 상태

- 앱명: `칭찬해줘`
- 서브카피: `지금 듣고 싶은 그 말`
- 버튼 5개: `응원해줘!`, `진정시켜줘!`, `위로해줘!`, `괜찮다고 해줘!`, `잘했다고 해줘!`
- 콘텐츠 구조: `quote-pack.v0.1.json`
  - `categories`: 5개
  - `releasePack`: 25개
  - `referencePack`: 25개
- 오디오 구조: `audio-manifest.v0.1.json`
  - 25개 asset slot
  - 현재 전부 `missing`
- UI: 홈 단일 화면, 5버튼, 현재 한마디 카드, fallback, 마지막 들은 한마디
- 플랫폼 준비: 로그인/결제/광고는 MVP stub으로 유지하되 Apps in Toss와 Google Play용 provider/store/network 후보를 명시

> 현재 6/10, 실제 오디오 25개가 붙으면 7/10, 반복 사용 루프까지 붙으면 8/10까지 갈 수 있다.

## gstack CEO 리뷰

### 유지해야 할 것

- 5버튼 구조
- 홈 단일 화면
- 버튼 탭 후 즉시 한마디 카드 갱신
- 사전 생성 로컬 오디오
- `releasePack`과 `audioManifest` 기반 데이터 로딩
- 오디오 실패 시 텍스트 fallback
- 알림/결제/계정/서버/AI 상담 제외
- `다시 듣기`와 `다른 한마디` 중심의 가벼운 조정 UX

이 제품은 복잡해질수록 약해진다. 사용자는 분석받고 싶어 하는 것이 아니라,
지금 자기 마음에 꽂히는 말을 듣고 싶어 한다.

### 바꿔야 할 해석

`칭찬해줘`를 "칭찬 문구 앱"으로 보면 상품성이 약하다.
더 정확한 해석은 다음이다.

> 내가 무너지는 순간에 누르는, 내 편 음성 버튼.

그러면 경쟁 범위도 명언 앱이 아니라 루틴/집중/멘탈케어 뒤에 붙는 회복 레이어가 된다.

### 가장 큰 상품성 리스크

1. 음성이 별로면 앱 전체가 무너진다.
2. 문장이 흔하면 `랜덤 위로 문구 앱`처럼 보인다.
3. 의료/상담처럼 보이면 정책과 신뢰 리스크가 커진다.
4. 명대사 IP를 그대로 쓰면 권리 리스크가 커진다.
5. 반복 사용 이유가 없으면 첫날 재미로 끝난다.

## Superpowers 브레인스토밍 리뷰

### 접근안 A: 지금 MVP 유지

내용:

- 5버튼
- 25문장
- 25오디오
- fallback
- 최근 들은 한마디

장점:

- 가장 빨리 제품의 본질을 검증한다.
- 오디오 품질이 좋은지 바로 판단할 수 있다.
- 구현 리스크가 낮다.

단점:

- 반복 사용 루프가 약하다.
- 문장/음성 품질이 부족하면 앱이 너무 작아 보인다.

판단:

> 추천. 지금은 이게 맞다.

### 접근안 B: 알림/저장/반응까지 추가

내용:

- 좋았어/별로야
- 저장
- 하루 1회 알림
- 알림 탭 후 해당 카테고리 재생

장점:

- D1/D7 반복 사용 이유가 생긴다.
- 어떤 문장이 먹히는지 알 수 있다.

단점:

- 오디오 검수 전 붙이면 핵심 실패 원인을 가린다.
- 알림 권한/플랫폼 처리가 MVP를 지저분하게 만든다.

판단:

> v0.2로 미룬다. 단, 데이터 구조만 준비한다.

### 접근안 C: 명대사/유료팩 중심

내용:

- 영화/드라마/명언 pack
- 유료 음성팩
- 유명 대사 중심 마케팅

장점:

- 훅은 강하다.
- 콘텐츠 판매 구조가 보인다.

단점:

- 권리 검수가 필요하다.
- 앱 본질 검증 전에 비용과 리스크가 커진다.
- 사용자는 "대사"보다 "지금 나에게 맞는 말"을 원할 수 있다.

판단:

> 지금은 보류. referencePack으로만 유지한다.

## 기획성 리뷰

### 좋은 점

- 버튼명이 사용자의 요청형이라 첫 화면에서 바로 이해된다.
- 감정/상황 조합을 버려서 피로도가 크게 줄었다.
- 오프라인 로컬 오디오라 비용 예측이 쉽다.
- 데이터팩 구조라 문장/오디오 운영이 가능하다.
- `칭찬해줘`라는 이름은 가볍고 행동을 유도한다.

### 애매한 점

- `칭찬해줘`는 앱명으로 좋지만, 실제 카테고리 중 `진정/괜찮음/위로`는 칭찬보다 넓다.
- `referencePack`에는 강한 후보가 있지만, `releasePack`은 아직 오리지널 문구 느낌이 남아 있다.
- 마지막 들은 한마디는 좋지만, 반복 사용의 핵심 장치가 되기에는 약하다.

### 반드시 고쳐야 할 점

- 문장 최종 검수는 텍스트 기준이 아니라 **음성으로 들었을 때 기준**으로 해야 한다.
- 음성 파일이 붙기 전까지는 상품성 판단을 유보해야 한다.
- 앱 설명/스토어 문구에서 `상담`, `치료`, `불안 개선`, `우울 완화` 같은 표현은 피해야 한다.

## 시장/정책 관점

Grand View Research는 글로벌 mental health apps 시장을 2024년 74.8억 달러, 2030년 175.2억 달러로
전망한다. 즉 큰 수요는 있다. 하지만 이 시장은 성장하는 만큼 신뢰/프라이버시/효능 검증 리스크도 크다.

Google Play는 개인/민감 데이터 처리 시 투명성, 목적 제한, 보안, 런타임 권한, 판매 금지 등을 요구한다.
Apple도 개인정보 보호와 데이터 최소화를 강하게 요구하고, 건강/의료 데이터는 특히 민감하게 다룬다.

따라서 `칭찬해줘`의 좋은 전략은 정신건강 치료 앱처럼 커지는 것이 아니라,
**개인정보를 거의 모으지 않는 가벼운 오디오 셀프케어 도구**로 남는 것이다.

## 상품화 전략

### v0.1

목표:

- "버튼 누르면 좋은 목소리로 바로 나온다" 검증

필수:

- 최종 25문장 검수
- 대표 5개 로컬 오디오 먼저 생성
- 5버튼 즉시 재생 경험 테스트
- 방향이 맞으면 나머지 20개 로컬 오디오 생성
- manifest path/status 연결
- 오디오 볼륨/길이/톤 QA
- 앱 설명에 비의료/비상담 톤 반영
- 로그인/결제/광고는 실제 SDK 없이 adapter stub으로만 준비

하지 말 것:

- 알림
- 결제
- 광고
- 계정
- AI 상담
- 실시간 TTS

## 수익화 판단

지금 광고는 비추천이다. 감정적으로 약한 순간에 광고가 뜨면 신뢰가 무너진다.

가능성 있는 수익화:

- 유료 음성 pack
- 프리미엄 화자 pack
- 시즌 pack
- 100문장 확장 pack
- 나중에 개인용 로컬 음성 import

다만 결제는 retention 검증 후에만 붙인다.

판단 기준:

- 7일 안에 3회 이상 재방문하는 사용자가 있는가?
- 같은 문장을 다시 듣는 비율이 충분한가?
- 사용자가 `저장`을 누르는 문장이 있는가?
- 사용자가 돈을 낼 만큼 "목소리" 차이를 느끼는가?

## 정책/권리 리스크

### 명대사 리스크

- `referencePack`에 유지 가능
- 기본 앱에는 노출하지 않음
- `releasePack`에는 low-risk 문장만 사용
- 유명 작품명/캐릭터/상표 연상은 스토어 문구에서도 조심

### 음성 리스크

- 실제 영화/드라마 음성 클립 금지
- 배우/캐릭터 성대모사 금지
- 중립 생성 음성 또는 라이선스 확보 음성만 사용
- ElevenLabs는 런타임 API가 아니라 사전 제작 도구로만 사용
- 출시용 오디오는 상업 사용 가능 조건을 충족하는 유료 플랜 생성물만 사용
- Beta 서비스 생성물이나 상업 사용 권한이 불명확한 파일은 출시본에서 제외

### ElevenLabs 제작 원칙

- 실제 오디오는 ElevenLabs에서 사전 생성한 로컬 파일을 사용한다.
- 앱 런타임에서 ElevenLabs API를 호출하지 않는다.
- API key나 ElevenLabs 계정 정보는 앱 번들에 넣지 않는다.
- 배우, 캐릭터, 유명인 성대모사는 하지 않는다.
- `audioManifest`는 제작 이력 관리를 위해 다음 필드를 기록할 수 있게 둔다.
  - `fileName`
  - `voiceId`
  - `model`
  - `generatedAt`
  - `durationMs`
  - `status`

### 정신건강 리스크

- 치료/상담/진단/개선 주장 금지
- 위기 상황 대응 앱처럼 보이지 않게 함
- 필요한 경우 "전문적인 도움이 필요한 경우 전문가나 긴급 지원을 이용하세요" 수준의 안내만 둠

## 다음 결정

가장 먼저 해야 할 일:

> 대표 5개 문장을 먼저 ElevenLabs로 생성하고, 앱에서 5버튼 즉시 재생 경험을 테스트한다.

그 다음에 판단할 것:

- 문장이 오디오로 들었을 때 충분히 좋은가?
- 버튼별 목소리 톤이 구분되는가?
- 텍스트 fallback 없이도 앱이 살아나는가?
- 같은 문장을 다시 듣고 싶은가?
- 나머지 20개 오디오를 같은 톤으로 확장해도 되는가?
- `이 카테고리 전체 보기`가 v0.2에 들어갈 만큼 직접 선택 니즈가 있는가?

## 최종 verdict

`칭찬해줘`는 "좋은 기획"이 되려면 기능을 더 붙이는 앱이 아니라,
한 번 눌렀을 때 사용자가 진짜로 멈칫하는 오디오 경험이 되어야 한다.

지금은 구현 범위를 더 넓히지 말고,

1. 대표 5문장 오디오 생성
2. 5버튼 즉시 재생 QA
3. 나머지 20개 오디오 생성
4. 버튼별 오디오 QA
5. 5명 정도에게 첫 사용 테스트
6. 그 뒤 `이 카테고리 전체 보기`를 v0.2 후보로 판단

이 순서가 맞다.

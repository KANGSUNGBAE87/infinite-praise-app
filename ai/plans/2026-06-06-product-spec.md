## 핵심 원칙 (설계 불변 규칙)

**모든 칭찬에는 영수증이 붙는다.**

- 근거 없는 칭찬은 한 마디도 내보내지 않는다
- "당신은 꾸준한 사람입니다 *(근거: 최근 30일 중 22일 기록)*" — 괄호가 신뢰의 장치
- 비교는 오직 과거의 나하고만. 남과 비교는 설계 단계에서 차단

---

## 타겟 사용자

**1차 타겟 (코어):**
- 20-30대 직장인
- "요즘 아무것도 못 하는 것 같다"는 감각을 자주 느끼는 사람
- 번아웃 초기, 무력감, 자존감 저하를 경험 중이지만 치료가 필요한 수준은 아닌 사람
- 이미 뭔가를 하고 있지만 그걸 스스로 인식하지 못하는 사람

**2차 타겟 (확장):**
- 자기계발에 관심 있지만 습관 트래커에 지친 사람
- 기록은 하고 싶지만 복잡한 앱이 부담스러운 사람

**안티-타겟 (만들지 않는 제품):**
- 완벽주의자가 성과를 더 체계적으로 관리하는 앱
- 남에게 자랑하거나 공유하는 앱
- 전문 멘탈헬스 치료를 대체하는 앱

## 칭찬의 3단계 모델

이게 핵심 알고리즘이고, 이 단계가 올라갈수록 앱의 가치가 올라간다.

| 단계 | 이름 | 예시 | 데이터 요건 |
|------|------|------|------------|
| 1단계 | 행동 반영 | "운동 14회 했네" | 기록 집계 |
| 2단계 | 패턴 탐지 | "몰아서 하는 게 아니라 꾸준한 사람이야" | 7일+ 분포 분석 |
| 3단계 | 정체성 서사 | "지치는 날에도 자신과의 약속을 지키는 사람이야" | 30일+ 교차 패턴 |

MVP는 1단계부터. 3단계는 데이터가 쌓인 이후 (D30+) 사용자에게 제공.

## 기능 정의

### MVP (앱인토스, 2-4주 목표)

**F1. 오늘의 기록 — 하루 한 줄**
- 오늘 한 일 1개를 자유 텍스트로 입력 (`"오늘 운동 30분"`, `"어려운 대화 했다"`)
- 카테고리 태그 자동 추천 (운동/일/관계/학습/휴식 — 5개)
- 입력은 5초 이내로 끝나야 함. 마찰 = 이탈

**F2. 오늘의 칭찬**
- 입력 직후 LLM이 근거 기반 칭찬 1개 생성
- 칭찬 형식: `[행동] + [의미/해석] + (근거: 실제 데이터)`
- 예: "오늘 운동을 했어. 피곤했을 텐데도 몸을 움직인 건 자신과의 약속을 지킨 거야. *(이번 달 8번째 운동)*"
- 빈 날 처리: 기록이 없으면 "오늘은 쉬는 날이야" + 이번 달 누적 보여주기

**F3. 주간 요약 (D7+)**
- 이번 주 기록 집계
- 패턴 한 줄: "이번 주는 주로 [카테고리]에 집중했네"
- 스트릭 표시 (연속 기록일)

**F4. 월말 리빌 (D30+)**
- 이달의 진짜 결산: 카테고리별 집계 + 눈에 띄는 패턴
- 타이틀: `"사실 당신은 생각보다 꾸준한 사람입니다"`
- 이달의 가장 인상적인 행동 1개 (LLM 선정 + 근거)
- 구체적인 숫자가 핵심: `"운동 14회, 코딩 22일, 독서 6권"`
- 리빌은 월초에 열림 (Spotify Wrapped 감성)

### V2 (독립 iOS 앱, 앱인토스 이후)

**F5. HealthKit 패시브 수집**
- 걸음 수, 수면 시간, 운동 시간 자동 수집
- 수동 기록에 자동 데이터를 보강 레이어로 추가
- "오늘 7,500보 걸었고, 코딩 2시간 했어" — 합쳐서 칭찬
- HealthKit 데이터만으로는 칭찬 미생성 (의미는 사용자가 직접 부여)

**F6. 스크린타임 연동**
- 유튜브/SNS 사용 감소를 성취로 전환
- "어제보다 유튜브 30분 덜 봤어" — 나쁜 습관 감소도 영수증 대상
- 베이스라인 필요: 첫 7일 평균을 기준점으로

**F7. 정체성 서사 (3단계 칭찬)**
- D30+ 사용자에게 활성화
- 행동 교차 분석: 바빴던 날에도 운동 기록 → "회복탄력성" 신호
- LLM 클러스터링으로 개인화 정체성 레이블 생성

### V3 (선택적, 추후 결정)
- 위젯 (홈화면 스트릭 / 오늘의 칭찬)
- Apple Watch 연동
- 기록 내보내기 (월간 PDF)

## "빈 날" 처리 규칙 (리텐션의 핵심)

빈 날을 어떻게 처리하느냐가 앱이 사는지 죽는지를 결정한다.

**규칙 3가지:**
1. **거짓말 금지** — 없는 걸 칭찬하면 신뢰 모델 전체가 무너짐
2. **줌아웃** — 오늘이 아니라 이번 달을 보여줌
3. **리프레이밍** — "14일 달리던 사람이 하루 쉰 건 실패가 아니라 회복이야"

**빈 날 UI 흐름:**
```
오늘 기록 없음
↓
"오늘은 쉬는 날. 이번 달 기록 보기"
↓
[이번 달 카테고리별 집계 표시]
↓
"운동 8회, 코딩 14일 — 어제의 나보다 여기까지 왔어"
```

## LLM 칭찬 생성 프롬프트 규칙

이걸 틀리면 앱 전체가 망가진다.

**강제 규칙 (절대 위반 금지):**
- 반드시 실제 데이터를 인용 (`*(이번 달 X번째)*`)
- 표현은 매번 다르게 변주 (같은 패턴 3회 이상 반복 금지)
- 남과 비교하는 표현 금지 ("평균보다", "보통 사람보다" 등)
- 없는 사실 날조 금지 (기록에 없는 행동을 칭찬하면 안 됨)
- 빈 날에는 "가장 작은 진실 찾기" (월간 누적 데이터로 대체)

**권장 어조:**
- 친구처럼, 하지만 데이터에 근거한
- 과장하지 않음 ("대단해요!"보다 "생각보다 꾸준한 사람이야")
- 정체성 언어 사용 (`"~하는 사람이야"`) — 습관보다 정체성이 더 강한 동기

## 기술 스택 결정 (MVP)

### 플랫폼: 앱인토스(Apps in Toss)란
앱인토스는 **토스 슈퍼앱(3,000만 유저) 안에 들어가는 미니앱 플랫폼**이다. 별도 앱 빌더가 아님.
- `npx create-ait-app`로 시작, **React Native + `@apps-in-toss/framework`** 기반 (WebView/Unity/Cocos도 선택 가능)
- 빌드 시 `.ait` 파일 생성 → 토스에 제출, 심사 4단계(운영·디자인·기능·보안) 2~3영업일
- 토스 로그인 / 앱인토스 인앱결제 SDK(구글·애플 영수증 검증 내장) 제공

### 핵심 아키텍처 원칙: 어댑터 패턴으로 이식성 확보
앱인토스 MVP → 구글플레이/앱스토어 독립 앱 이식을 위해, **처음부터 플랫폼 의존 부분을 인터페이스로 분리**한다.

| 레이어 | 앱인토스 | 구글플레이 독립 앱 | 이식성 |
|--------|----------|-------------------|--------|
| 비즈니스 로직 (기록/스트릭/집계) | 순수 TypeScript | 순수 TypeScript | ✅ 100% 공유 |
| UI 컴포넌트 | React Native | React Native | ✅ 거의 공유 |
| `AuthAdapter` | 토스 로그인 | Google/Firebase Auth | ⚠️ 구현체 교체 |
| `PaymentAdapter` | 앱인토스 IAP SDK | Google Play Billing (`react-native-iap`) | ⚠️ 구현체 교체 |
| `StorageAdapter` | 앱인토스 저장소 | AsyncStorage/SQLite | ⚠️ 구현체 교체 |
| LLM 호출 | 백엔드 프록시 | 백엔드 프록시 | ✅ 100% 공유 |

→ 코드 80~90%를 한 번만 작성해 두 플랫폼에 출시.

### LLM 호출: 백엔드 프록시 필수 (⚠️ 클라이언트 직접 호출 금지)
**API key를 앱 클라이언트에 절대 넣지 않는다.** RN 번들·네트워크 트래픽에서 추출되면 제3자가 무제한 호출 → 비용 폭탄.

```
[앱(앱인토스/구글플레이)] --(사용자 인증 토큰)--> [백엔드 프록시] --(x-api-key)--> [Claude API]
                                                      ↑ API key는 여기 환경변수에만 존재
```
- **백엔드**: Cloudflare Workers / Vercel Functions / Supabase Edge Functions (무료 티어로 MVP 충분)
- 백엔드 책임: API key 은닉, 사용자 인증, rate limiting, 남용 차단, 결제 영수증 검증
- **온디바이스 소형 모델은 채택 안 함**: 한국어 칭찬 품질이 Haiku보다 낮아 "영수증 기반 칭찬" 신뢰가 무너지고, 앱 용량 폭증으로 미니앱 제약과 충돌

### LLM 모델 및 비용
- **모델**: `claude-haiku-4-5` (input $1.00 / output $5.00 per 1M tokens)
- 칭찬 1건 ≈ 0.001달러(약 1.5원) · 사용자 1명/월(30회) ≈ 45원 · 1,000명/월 ≈ 45,000원
- → 무료 제공도 부담 없음. 월 2,900원 구독 1건이 LLM 비용 64개월치 커버.

### 데이터 구조 (StorageAdapter 경유)
```json
{
  "date": "2026-06-06",
  "entry": "운동 30분",
  "category": "운동",
  "praise": "...",
  "source_data": { "streak": 8, "monthly_count": 14 }
}
```

### 수익화 / 인증 / 광고 설계 (미리 결정)
- **인증**: 앱인토스=토스 로그인(IAP 전제), 구글플레이=Google/Firebase Auth — `AuthAdapter`로 추상화
- **인앱결제**: 영수증 검증은 두 플랫폼 다 백엔드에서 (클라이언트 검증은 위조 가능) → LLM 프록시 백엔드에 검증 엔드포인트 통합
- **광고**: 앱인토스는 외부 광고 SDK 제약 가능성 ↑ → **구독(IAP) 우선, 광고는 구글플레이 버전 AdMob에서**

### 독립 앱 V2 확장 (HealthKit 필요 시)
- React Native 유지 → `react-native-health`(HealthKit), Screen Time은 네이티브 모듈 필요
- 동일한 백엔드 프록시·어댑터 구조 재사용

## 수용 기준 (Acceptance Criteria)

**기록 입력:**
1. 텍스트 입력 → 저장까지 UI 인터랙션 ≤ 3탭
2. 카테고리 자동 태그 추천 정확도 ≥ 70% (수동 샘플 20개 기준)
3. 오프라인에서도 기록 저장 가능 (칭찬 생성은 온라인 시에만)

**칭찬 생성:**
4. 칭찬 텍스트에 항상 실제 데이터 인용 포함 (`*(이번 달 N번째)*` 형식)
5. 동일 패턴 3회 연속 반복 없음 (20개 연속 테스트)
6. "남과 비교"하는 표현 0건 (20개 샘플)
7. 칭찬 생성 응답 시간 ≤ 3초 (네트워크 정상 시)

**빈 날 처리:**
8. 기록 없는 날 → "빈 날 UI"가 표시됨 (정상 칭찬 UI 아님)
9. 빈 날 UI에서 이번 달 누적 데이터가 표시됨
10. 빈 날 칭찬 텍스트에 날조된 행동 없음

**월말 리빌:**
11. 기록이 1개 이상인 달에만 리빌 생성
12. 리빌에 카테고리별 정확한 집계 포함
13. 리빌 텍스트에 구체적 숫자 포함 (`"운동 14회"` 형식)

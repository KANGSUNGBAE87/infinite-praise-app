# 칭찬해줘 알림형 MVP 최종 기획안

## 2026-07-04 현재 제품 기준

앱의 현재 사용자-facing 이름은 `내편한마디`이며, 영어명은 `A Word for Me`다. 기존 `칭찬해줘`는 기능 모드명으로 남고, `잔소리해줘`와 함께 첫 핵심 선택지가 된다.

v0.6 구현 기준 핵심 흐름:

1. 사용자는 `칭찬해줘`, `잔소리해줘`, `직접 쓸게` 중 하나를 고른다.
2. `칭찬해줘`/`잔소리해줘`는 상황, 감정, 톤, 강도와 120자 이내 맥락을 조합해 후보를 만든다.
3. `직접 쓸게`는 사용자가 받을 문장을 직접 쓰고, 로컬 안전검사 통과 후 바로 알림 시간 설정으로 이동한다.
4. 앱은 `/api/generate-candidates` 서버 프록시로 DeepSeek V4 Pro 후보 5개 생성을 시도한다.
5. AI 프록시가 없거나 실패하면 로컬 저장/템플릿 후보를 사용자에게 보여주지 않고, AI 연결 필요 안내만 표시한다.
6. 모든 AI 후보에는 `AI가 만든 후보` 라벨과 신고 액션을 붙인다.
7. 사용자가 후보를 고르고 직접 수정한 뒤 받을 시간을 하나 이상 정한다.
8. 브라우저 Notification API가 가능한 환경에서는 저장한 모든 시간에 실제 알림 예약을 시도하고, 불가능하면 앱 안 미리보기로 저장한다.

AI/알림 정책 기준:

- AI provider key는 클라이언트에 넣지 않는다. 서버 전용 `DEEPSEEK_API_KEY`만 사용한다.
- AI 입력과 후보 draft는 localStorage에 장기 저장하지 않는다. 사용자가 최종 선택/수정한 문장만 앱 상태로 남긴다.
- 이름, 연락처, 이메일, 건강/진단/치료 관련 단어는 AI 전송 전 마스킹한다.
- 자해, 폭력, 혐오, 의료/상담, 모욕/수치심 패턴은 입력 전/출력 후 모두 차단하거나 경고한다.
- 신고는 후보 원문 없이 후보 id, 모드, 스타일, source, locale, reason만 서버에 보낸다. Supabase service-role 설정이 있을 때 `praise_candidate_reports`에 저장한다.
- GitHub Pages는 정적 호스팅이므로 `/api/*` 프록시는 실행되지 않는다. 실제 DeepSeek/신고 운영은 Supabase Edge Function, Vercel, Netlify, Cloudflare Worker 같은 서버 프록시 배포가 필요하다.
- AI 실패/무응답 시 `message-templates.v0.1.json` 로컬 템플릿을 사용자-facing 후보로 대체하지 않는다. 템플릿 카탈로그는 taxonomy seed, 테스트, 향후 CMS 이관 기준으로만 유지한다.
- Apps in Toss 실푸시는 아직 클라이언트에서 직접 완료할 수 없다. 실제 Toss 알림은 Smart Message 서버, mTLS, 템플릿 검수, 알림 동의가 필요하다.
- Apps in Toss 실푸시 문구는 검수된 일반 문구 + 앱 내부 개인화 화면 deep link 방식이 안전하다.

문구 고정 문제 해결 기준:

- 문구는 코드 안 5개 배열이 아니라 `message-templates.v0.1.json`의 template catalog를 우선한다.
- 기본 축은 `mode + situation + feeling + tone + intensity + purpose`다.
- 상황 축: 공부, 일, 운동, 집안일, 수면, 폰 내려놓기.
- 감정 축: 지침, 미룸, 불안, 뿌듯함, 무기력/막힘.
- 톤 축: 다정, 차분, 현실적, 짧고 직설, 유머 약간.
- 강도 축: 부드럽게, 정확하게, 단호하게.
- 현재 후보 생성 경험은 AI 서버 프록시가 연결된 경우에만 제공한다. 비AI 경로는 `직접 쓸게`로 분리하고, 로컬 템플릿은 향후 검수/원격 설정/CMS 이관용 seed로만 유지한다.

## 1. 최종 판단

`칭찬해줘` v0.1은 **음성 즉시재생 앱**이 아니라 **알림형 자기 조율 메시지 앱**으로 간다.

기존의 "버튼을 누르면 좋은 목소리로 바로 한마디가 나온다"는 경험은 매력적이지만, 앱을 반복해서 쓰게 만드는 습관 루프가 약하다. 반대로 알림형 MVP는 사용자가 직접 시간과 메시지를 정하기 때문에 일상 속 재방문 이유가 생긴다.

최종 핵심 약속:

> 내가 정한 시간들에, 내가 고른 톤으로, 지금 나에게 필요한 말을 받아본다.

한 줄 정의:

> `칭찬해줘`는 사용자가 원하는 말투의 한마디를 고르고, 자기가 정한 시간에 알림으로 받아보는 자기 조율 메시지 앱이다.

## 2. 제품 본질

이 앱은 단순 문구 앱, 명언 앱, 상담 앱이 아니다.

`칭찬해줘`는 사용자가 반복적으로 흔들리는 순간에 스스로에게 보내는 **내 편 메시지 / 행동 리마인드 앱**이다.

제품의 본질은 세 가지다.

1. 지금 나에게 필요한 말투를 고른다.
2. 내가 원하는 시간에 받는다.
3. 받은 메시지를 보고 다시 움직이거나, 스스로를 다정하게 인정한다.

## 3. 포지션

정확한 포지션:

> 칭찬과 잔소리를 내가 원하는 시간에 받는 가벼운 자기 조율 알림 앱

사용 가능한 표현:

- 짧은 응원
- 다정한 한마디
- 내 편이 되어주는 메시지
- 해야 할 일을 살짝 밀어주는 잔소리
- 내가 정한 시간에 받는 칭찬과 리마인드
- 하루 중 여러 번 받아보는 내 편 한마디
- 하루를 다시 움직이게 하는 한마디

피해야 할 표현:

- 치료
- 상담
- 진단
- 우울 개선
- 불안 개선
- 공황 완화
- 정신건강 치료
- 의학적 효과
- 전문가 수준 케어

정책 판단:

- 비게임 생활형 앱으로 본다.
- 의료, 상담, 진단 효과를 주장하지 않는다.
- 알림 메시지는 자기 조율과 리마인드로 표현한다.
- Apps in Toss 출시 가능성을 유지하되, 알림 권한과 개인정보 수집은 기능 필요 시점에 최소화한다.

## 4. 앱명

v0.1 앱명은 `칭찬해줘`를 유지한다.

이유:

- 쉽고 기억하기 좋다.
- 행동을 유도하는 이름이다.
- 따뜻한 앱으로 먼저 인식된다.
- 기존 기획과 구현 흐름을 유지할 수 있다.

다만 내부 기능은 칭찬만이 아니라 세 가지 모드로 확장한다.

- 칭찬
- 잔소리
- 직접 쓰기

장기적으로 잔소리 사용률이 높고 서비스가 칭찬보다 넓어지면 v0.2 이후 상위 앱명 변경을 다시 검토한다.

후보:

- 내 편 한마디
- 지금 한마디
- 말해줘
- 한마디 알림
- 나한테 한마디

## 5. v0.1 핵심 모드

### 5.1 칭찬

용도:

- 인정받고 싶을 때
- 지쳤을 때
- 오늘을 버텼다는 말을 듣고 싶을 때
- 자기 전에 스스로를 다정하게 마무리하고 싶을 때

톤:

- 다정함
- 인정
- 위로
- 부담 없는 응원

문구 원칙:

- 가짜 확신을 피한다.
- "넌 최고야"만 반복하지 않는다.
- 사용자가 실제로 해낸 것, 버틴 것, 다시 시작할 수 있는 여지를 인정한다.

예시:

- 오늘 버틴 것만으로도 잘했어.
- 지금 한 걸음이면 충분해.
- 쉬어도 네가 해온 일은 사라지지 않아.
- 완벽하지 않아도 오늘의 너는 충분해.
- 아무도 몰라도 네가 해낸 건 사라지지 않아.

### 5.2 잔소리

용도:

- 미루는 일을 시작해야 할 때
- 폰을 내려놔야 할 때
- 루틴을 지켜야 할 때
- 누가 살짝 밀어줬으면 할 때
- 약속한 일을 시작하게 만들고 싶을 때

톤:

- 짧음
- 직설적
- 행동 유도
- 현실적
- 모욕, 비난, 수치심 금지

좋은 잔소리 원칙:

- 사람을 평가하지 않는다.
- 행동만 짚는다.
- 다음 행동을 제시한다.
- 길게 설명하지 않는다.
- 죄책감보다 실행감을 준다.

금지 예시:

- 넌 왜 맨날 이러냐.
- 이러니까 네가 안 되는 거야.
- 한심하다.
- 너는 의지가 약해.
- 살 빼야지.
- 정신 차려, 네 인생 망한다.

허용 예시:

- 폰 내려. 지금은 네가 정한 시간이다.
- 완벽하게 말고, 일단 열어.
- 하기 싫은 건 알겠고, 그래도 5분만 하자.
- 지금 시작 안 하면 또 밀린다. 하나만 하자.
- 생각은 충분히 했고, 이제 손을 움직이자.

잔소리 강도:

- v0.1: 부드럽게, 정확하게
- v0.2 후보: 세게

`세게`는 반응 데이터와 문구 검수 없이 넣지 않는다.

### 5.3 직접 쓰기

용도:

- 사용자가 직접 받을 말을 입력한다.
- 기존 메시지를 수정해 자기 말투로 만든다.
- 특정 목표나 루틴에 맞는 문장을 저장한다.

예시:

- 퇴근하면 바로 운동복 입기.
- 밤 11시에는 폰 내려놓기.
- 아침에는 완벽하게 말고 물 한 잔부터.
- 오늘도 나를 너무 몰아붙이지 않기.

MVP 범위:

- 메시지 직접 입력
- 기존 메시지 복사 후 수정
- 모드 선택
- 알림 시간 연결
- 저장

MVP 제외:

- 클라이언트 직접 AI 호출
- 의료/상담/진단 성격의 AI 답변
- 실시간 TTS
- 계정 동기화
- 공유 템플릿 마켓

## 6. v0.1 사용자 흐름

### 6.1 첫 사용 흐름

1. 사용자가 모드를 고른다.
   - 칭찬
   - 잔소리
   - 직접 쓰기
2. 메시지 후보를 본다.
3. 마음에 드는 메시지를 선택하거나 수정한다.
4. 알림 시간을 고른다.
5. 알림 미리보기 방식을 고른다.
   - 메시지 그대로 표시
   - 앱에서만 보기
6. 저장한다.
7. 정해진 시간에 알림을 받는다.
8. 알림을 탭하면 앱에서 메시지 상세를 본다.

### 6.2 반복 사용 흐름

1. 오늘 받을 메시지를 본다.
2. 메시지가 좋으면 계속 사용한다.
3. 별로면 문구를 수정하거나 다른 메시지로 바꾼다.
4. 시간이 맞지 않으면 시간을 바꾼다.
5. 필요 없으면 알림을 끈다.

v0.2 후보 반응:

- 좋았어
- 별로야
- 수정하기
- 이 시간 괜찮아
- 다른 시간에 받기

## 7. 화면 구조

### 7.1 홈

홈의 역할:

- 오늘 받을 메시지를 보여준다.
- 활성화된 알림을 보여준다.
- 새 한마디 알림을 만들게 한다.

필수 요소:

- 앱명: 칭찬해줘
- 서브카피: 내가 정한 시간에 받는 칭찬과 잔소리
- 오늘 받을 한마디
- 활성화된 알림 목록
- 새 한마디 알림 만들기 CTA

예시:

```text
칭찬해줘
내가 정한 시간에 받는 칭찬과 잔소리

오늘 받을 한마디
오후 2:00  폰 내려. 지금은 네가 정한 시간이다.
밤 11:00   오늘 버틴 것만으로도 잘했어.

활성화된 알림
[칭찬] 매일 밤 11:00
[잔소리] 평일 오후 2:00

[새 한마디 알림 만들기]
```

### 7.2 새 알림 만들기

흐름:

1. 모드 선택
2. 메시지 선택 또는 직접 작성
3. 알림 표시 방식 선택
4. 시간 선택
5. 저장 전 미리보기

모드 선택 UI:

- 칭찬: 다정하게 인정받고 싶을 때
- 잔소리: 지금 해야 할 일을 시작하고 싶을 때
- 직접 쓰기: 내가 받을 말을 직접 정하고 싶을 때

### 7.3 메시지 선택

칭찬/잔소리 모드는 기본 메시지를 보여준다.

요구사항:

- 메시지는 카드 형태로 보여준다.
- 선택 후 수정할 수 있어야 한다.
- 잔소리는 강도 칩을 제공한다.
  - 부드럽게
  - 정확하게
- 강도별로 문구가 달라져야 한다.

### 7.4 시간 선택

시간 설정은 사용자가 가장 귀찮아할 지점이므로 프리셋 중심으로 간다.

추천 프리셋:

- 아침 시작
- 점심 전
- 오후 늘어질 때
- 퇴근 전
- 자기 전

반복 옵션:

- 매일
- 평일
- 주말
- 요일 직접 선택

미세 조정:

- 10분 당기기
- 10분 늦추기
- 직접 입력

### 7.5 저장 전 미리보기

저장 전에는 사용자가 실제 알림을 상상할 수 있어야 한다.

표시 예시:

```text
내일 오후 2:00
폰 내려. 지금은 네가 정한 시간이다.

반복: 평일
표시: 메시지 그대로 표시

[저장하기]
```

## 8. v0.1 데이터 구조

문구와 알림은 UI 컴포넌트에 하드코딩하지 않는다.

```ts
type MessageMode = "praise" | "nag" | "custom";
type NagIntensity = "soft" | "direct";
type NotificationDisplayMode = "fullText" | "private";
type RepeatRule = "daily" | "weekday" | "weekend" | "customDays";

type MessageTemplate = {
  id: string;
  mode: MessageMode;
  intensity?: NagIntensity;
  title?: string;
  displayText: string;
  notificationText: string;
  tags: string[];
  status: "release" | "draft";
};

type Reminder = {
  id: string;
  mode: MessageMode;
  messageTemplateId?: string;
  customText?: string;
  scheduledTime: string;
  repeatRule: RepeatRule;
  customDays?: number[];
  displayMode: NotificationDisplayMode;
  enabled: boolean;
  createdAt: string;
  updatedAt: string;
};
```

초기 저장:

- v0.1 개발/웹 MVP는 localStorage 기반으로 시작한다.
- 앱인토스/Google Play 포팅 시 StorageAdapter 뒤로 이동한다.
- 알림 예약도 NotificationAdapter 뒤에 둔다.

## 9. 플랫폼/어댑터 원칙

v0.1은 로그인, 결제, 광고를 실제 구현하지 않는다. 다만 이미 있는 플랫폼 어댑터 원칙은 유지한다.

준비할 어댑터:

- AuthAdapter
- PaymentAdapter
- AdsAdapter
- StorageAdapter
- AnalyticsAdapter
- NotificationAdapter

NotificationAdapter 초안:

```ts
type NotificationAdapter = {
  getPermissionStatus(): Promise<"granted" | "denied" | "prompt" | "unsupported">;
  requestPermission(): Promise<"granted" | "denied" | "unsupported">;
  scheduleReminder(reminder: Reminder): Promise<{ ok: boolean; platformId?: string; reason?: string }>;
  cancelReminder(reminderId: string): Promise<{ ok: boolean; reason?: string }>;
};
```

중요:

- 웹 브라우저만으로는 앱이 닫힌 뒤 정해진 시각에 알림을 안정적으로 보장하기 어렵다.
- 따라서 v0.1 웹 구현은 "알림 설계/저장/미리보기"까지 만들고, 실제 안정 알림은 앱 플랫폼의 NotificationAdapter에서 완성한다.
- 사용자에게 출시 전까지 "반드시 알림이 온다"고 과장하지 않는다.

## 11. 음성의 위치

음성은 폐기하지 않는다. 다만 v0.1의 핵심 기능에서 제외한다.

이유:

- 알림형 앱에서는 메시지/시간/반복 루프가 먼저 검증되어야 한다.
- 음성을 먼저 만들면 문구와 사용 순간이 바뀔 때 비용이 커진다.
- 사용자가 실제로 저장하고 반복해서 받는 문장을 확인한 뒤 음성화하는 편이 안전하다.

v0.2 이후 음성 적용 후보:

- 사용자가 자주 저장한 문구
- 반응이 좋은 기본 템플릿
- 자기 전 칭찬 메시지
- 아침 시작 메시지

원칙:

- 앱 런타임에서 TTS API를 호출하지 않는다.
- 사전 생성한 로컬 오디오만 사용한다.
- 배우/캐릭터/유명인 성대모사는 하지 않는다.
- 상업 사용 가능 권리가 확인된 오디오만 쓴다.

## 12. 성공 기준

v0.1의 성공 기준은 "좋은 목소리"가 아니라 "사용자가 직접 다시 설정하고 싶어 하는가"다.

핵심 지표 후보:

- 첫 알림 생성 완료율
- 첫 알림 생성까지 걸리는 시간
- 활성 알림 유지율
- 알림 수정률
- 알림 끄기율
- 직접 쓰기 사용률
- 잔소리 모드 선택률
- 메시지 수정률

정성 질문:

- 이 말을 이 시간에 받는 게 의미 있었나?
- 문구를 바꾸고 싶었나?
- 잔소리가 불쾌했나, 도움이 됐나?
- 시간을 다시 설정하기 쉬웠나?

## 14. 최종 실행 방향

최종적으로 v0.1은 다음 방식으로 기획한다.

1. 앱명은 `칭찬해줘`로 유지한다.
2. 기능의 중심은 `칭찬`, `잔소리`, `직접 쓰기` 세 모드다.
3. 사용자는 메시지를 고르거나 수정하고, 받을 시간을 하나 이상 정한다.
4. 홈은 오늘 받을 메시지와 활성 알림을 중심으로 구성한다.
5. 음성은 v0.1에서 제외하고, 반응 좋은 문구를 확인한 뒤 v0.2 이후 로컬 오디오로 붙인다.
6. 로그인, 결제, 광고는 stub으로 유지한다. AI는 서버 프록시로만 붙이고, 실패 시 후보를 만들지 않는다.
7. 의료/상담/치료 앱처럼 보이지 않게 표현과 문구를 통제한다.
